# 异常处理机制 

 程序运行时出错叫“异常”。异常分类体系
 
```
Throwable（所有异常和错误的父类）
├── Error（系统错误，程序无法处理）
└── Exception（应用级异常，程序可以捕获处理）
    ├── RuntimeException（运行时异常 / 非受检异常）
    │   
    └── 其他 Exception（编译时异常 / 受检异常）
```

运行时异常 / 非受检异常：编译不检查，不强制处理
编译时异常 / 受检异常：必须显式 try-catch 或 throws


常见的**RuntimeException**（运行时异常）：
- `NullPointerException`：对一个 `null` 值调用了方法或属性
- `ArrayIndexOutOfBoundsException`：数组索引越界
- `ArithmeticException`：数学运算错误（如除以零）
- `ClassCastException`
- `NumberFormatException`

常见的非**RuntimeException**（编译时异常）：
- `IOException`
- `FileNotFoundException`
- `SQLException`
- `ClassNotFoundException`

## `try-catch`

如果不处理受检异常，程序会直接崩溃。`try-catch` 用来捕获异常，让程序即使出错也能继续运行或给出友好提示。

**基本语法：**

```java
try {
    // 可能产生异常的代码
} catch (异常类型 变量名) {
    // 异常处理代码
}
```

**try-catch-finally 语法**

```java
try {
    // 可能出异常的代码
    语句1;
    语句2;  // 这里抛异常
    语句3;  // ← 永远不会执行
} catch (异常类型 变量名) {
    // 异常处理
} finally {
    // 无论是否异常，都会执行（可选）
}
继续执行后续代码;
```
**执行流程口诀**：
1. try 执行，正常则跳过 catch 进入 finally，然后结束。
2. try 异常则跳入匹配的 catch，catch 执行完进入 finally，然后结束。
3. 不管是 return、异常、正常结束，finally 一定插在最后一步前执行。

**关键规则**

- 单独 `try` 不能存在，后面必须有至少一个 `catch` 或一个 `finally`。
- 异常抛出后，`try` 块剩余代码**全部跳过**，不可逆
- 多个 `catch` 可以连续写，分别捕获不同异常类型，匹配到第一个符合条件的就执行，后面的 `catch` 不再执行。
-  `catch` 执行完后，从 `catch` 后面继续往下走
- 子类异常 `catch` 必须写在父类异常 `catch` 前面，否则编译错误（子类被父类覆盖，后面的不可达）。
- `finally` 块无论是否异常都会执行
- 被捕获的异常不会导致程序崩溃


`finally` 块的作用及特点

**规则**：
① `finally`中的代码总是会执行（除`System.exit(0)`等极端情况）。  
② 即使`try`或`catch`中有`return`，`finally`也在`return`前执行。
③`finally`中若有`return`会覆盖`try`/`catch`中的`return`值。
④`finally` 不是必须的，`try-catch`、`try-finally`、`try-catch-finally` 三种组合都合法。

## `throw`与`throws`

| 关键字      | 位置        | 作用             |
| -------- | --------- | -------------- |
| `throw`  | 方法**内部**  | 实际抛出异常对象       |
| `throws` | 方法**签名上** | 声明本方法可能抛出的异常类型 |

- `throw`：用在方法体内，手动抛出一个异常对象，如`throw new Exception()`。  
- `throws`：用在方法声明尾部，声明该方法可能抛出哪些异常类，多个用逗号分隔。  
- 受检异常必须遵循"捕获或声明"原则：要么`try-catch`处理，要么`throws`声明。
